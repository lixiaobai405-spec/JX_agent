### PDCA 循环业务逻辑概述

本系统基于 PDCA（Plan-Do-Check-Act）循环理论设计，实现完整的绩效管理闭环。PDCA 是一种持续改进的管理方法，通过四个阶段的循环迭代，不断提升绩效管理的质量和效果。

**PDCA 循环定义**：
- **P（Plan-计划）**：制定绩效目标和实施计划
- **D（Do-执行）**：执行计划并记录过程数据
- **C（Check-检查）**：评估执行结果与目标的差异
- **A（Act-行动）**：根据评估结果采取改进措施

**四个阶段的核心功能**：
1. **P 阶段 - 智能定标引擎**：通过 AI 分析岗位特征，自动生成个性化的绩效指标和目标
2. **D 阶段 - 智能追踪与反馈**：实时监控执行进度，提供诊断报告和改进建议
3. **C 阶段 - 智能评估与定级**：多维度考核评价，自动计算得分并生成等级
4. **A 阶段 - 个人复盘与发展**：分析绩效结果，制定发展计划，为下一周期提供改进建议

**阶段间的流转关系**：
- P → D：绩效合约确认后，触发执行追踪阶段
- D → C：考核周期结束时，触发考核评估阶段
- C → A：最终结果确认后，触发复盘发展阶段
- A → P：目标继承建议连接下一周期的计划阶段

**闭环机制**：每个周期的 A 阶段输出（发展计划、改进建议）会作为下一周期 P 阶段的输入，形成持续改进的螺旋上升循环。

### P 阶段（Plan-计划）- 智能定标引擎

**核心流程**：岗位扫描 → 智能归类 → 策略加载 → 指标生成 → 目标设定

**5种岗位原型**：
- **S 类（铁军型）**：销售/业务岗位，高量化、短周期、结果导向
- **P 类（项目型）**：研发/设计岗位，低量化、长周期、创造性工作
- **O 类（运营型）**：生产/客服岗位，高量化、短周期、标准化流程
- **F 类（职能型）**：职能/支持岗位，低量化、短周期、服务响应
- **M 类（管理型）**：高管/负责人，低量化、长周期、战略规划

**3维度评分**（AI 评分 1-10）：
- **可量化度**：数据可得性（高=数据定义，低=主观评价）
- **产出周期**：工作频率（短=天/周，长=项目/年）
- **工作性质**：标准化程度（标准化=SOP，创造性=无标准）

**8维度策略矩阵**：
1. 指标来源（组织目标分解/岗位职责/关键任务/客户诉求）
2. 评价人（上级/360度/客户/客观数据）
3. 指标属性（定量/定性/个人/团队）
4. 指标维度（业绩结果/过程行为/能力态度）
5. 目标值设定（自上而下/历史推导/外部标杆/自我设定）
6. 考核周期（月/季/半年）
7. 辅导周期（频繁细致/关键节点）
8. 结果应用（短期奖励/晋升/持续改善）

### D 阶段（Do-执行）- 智能追踪与反馈

**核心流程**：填报推送 → 实时分析 → 反馈报告

**达成率计算**：
- **正向指标**（越高越好）：达成率 = (实际值 / 目标值) × 100%
- **反向指标**（越低越好）：偏差值 = 目标值 - 实际值（正偏差=优秀，负偏差=滞后）
- **定性指标**：状态映射（已完成=100%，进行中=50%，延期=0%）

**进度偏差校验**：
- 综合进度偏差 = 加权达成率 - 时间进度
- 示例：任务完成 40%，时间过去 60%，偏差 = -20%（严重滞后）

**AI 诊断报告**：
- **红绿灯清单**：🟢优势项（>100%）、🟡预警项（80%-100%）、🔴风险项（<80%）
- **根因分析**：基于岗位特征和数据逻辑，提供诊断假设
- **改进建议**：具体可执行的下一步行动，支持一键申请上级辅导

---

### C 阶段（Check-考核）- 智能评估与定级

**核心流程**：自评 → 他评 → 算分定级 → 确认归档

**多源评价**：
- **自评**：员工确认实际数据并进行自我评分
- **上级评价**：直接上级核实数据并最终打分
- **360度评价**：同事、下属、跨部门协作者评分
- **客户评价**：内外部客户满意度评分

**等级规则**：
- **绝对评价**：根据分数段直接匹配等级（如 ≥90分=S级）
- **相对评价（强制分布）**：按部门排名分配等级（如 20% S级、70% B级、10% C级）

**红线指标一票否决机制**：
- 触碰红线指标（如安全事故、重大投诉）直接降级或评为不合格
- 红线指标设定为扣分项或一票否决，不占基础权重

### A 阶段（Act-行动）- 个人复盘与发展

**核心流程**：复盘报告 → 发展计划 → 新周期继承

**AI 解读绩效结果**：
- **优势分析**：提取高分指标，识别核心能力
- **待改进项**：提取低分指标，明确发展方向
- **分级建议**：针对 S/A/B/C 不同等级，提供差异化的发展建议

**SMART 原则校验**：
- **Specific（具体）**：目标是否明确清晰
- **Measurable（可衡量）**：是否有量化标准
- **Achievable（可达成）**：是否符合实际能力
- **Relevant（相关）**：是否与绩效改进相关
- **Time-bound（有时限）**：是否设定完成时间

**目标继承建议**（连接下一周期 P 阶段）：
- 在新周期 P 阶段的指标生成环节，AI 自动提醒员工上一周期的发展计划
- 员工可选择将发展计划转化为新周期的正式绩效指标
- 确保持续改进的闭环连接

### 阶段流转与数据依赖

**P → D 流转**：
- **触发条件**：performance_contract 表中的合约状态变更为"已确认"
- **数据流转**：从 performance_contracts 和 indicators 表读取目标数据，生成 checkin_tasks 填报任务

**D → C 流转**：
- **触发条件**：periods 表中的考核周期结束（end_date 到期）
- **数据流转**：从 checkin_records 表汇总执行数据，生成 self_assessments 自评任务和 evaluation_tasks 评价任务

**C → A 流转**：
- **触发条件**：final_results 表中的最终结果状态变更为"已确认"
- **数据流转**：从 final_results、evaluations、score_aggregates 表读取考核数据，生成 review_reports 复盘报告

**A → P 流转**：
- **触发条件**：新考核周期开始，进入 P 阶段的指标生成环节
- **数据流转**：从 inheritance_suggestions 表读取上一周期的改进建议，作为新周期指标生成的参考输入

## 数据库设计原则

### 1. 命名规范
- 表名：复数形式，小写+下划线（如：users, evaluation_tasks）
- 字段名：小写+下划线（如：user_id, created_at）
- 外键：关联表单数_id（如：user_id, goal_id）

### 2. 通用字段
- id：VARCHAR(36) PRIMARY KEY（UUID）
- created_at：TIMESTAMP NOT NULL
- updated_at：TIMESTAMP NOT NULL
- deleted_at：TIMESTAMP NULLABLE（软删除）

### 3. 索引策略
- 主键自动索引
- 外键字段建立索引
- 频繁查询字段建立索引
- 唯一约束字段建立唯一索引

### 4. 数据完整性
- 外键约束确保引用完整性
- 枚举类型限制字段取值范围
- NOT NULL约束防止空值
- CHECK约束验证数据有效性（如权重范围0-1）

### 5. 性能优化
- JSON字段存储复杂结构，减少表数量
- 合理使用索引，避免过度索引
- 软删除保留历史数据
- 分区表处理大数据量（如audit_logs按时间分区）
